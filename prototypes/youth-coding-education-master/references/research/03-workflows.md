### 1. 设计一堂 Scratch 启蒙课

- **One-liner**: 为6-9岁儿童设计一节60分钟 Scratch 项目式编程课，产出课件+学生项目模板。
- **Trigger**: 新学期开课、新班级入学、公开体验课需要。
- **Output**: 60分钟教案（含PPT/板书）、Scratch starter 项目文件、评价量规。
- **入门 SOP**:
  1. 确定教学主题与目标: 选一个与孩子兴趣相关的场景（动物游戏、节日故事、迷宫），确定1-2个核心知识点（如事件+循环），写出"学生能做到…"的可观察目标；inputs: 课程大纲/年级、outputs: 教学目标陈述。
  2. 拆解知识点与认知阶梯: 参照 CSTA K-12 标准 (evidence: [T03-S004]) 将目标分解为「看懂→模仿→改动→创作」四阶，每阶设计对应活动；inputs: 教学目标，outputs: 活动序列草稿。
  3. 设计 Scratch 项目载体: 在 scratch.mit.edu (evidence: [T03-S001]) 制作 starter 项目，留出3-5个关键空白让学生填写；inputs: 主题，outputs: starter .sb3 文件。
  4. 准备教学素材: 录制或收集演示视频（MIT Learning Library evidence: [T03-S002]）、准备问题卡、打印"调试检查表"；inputs: starter项目，outputs: 课堂素材包。
  5. 撰写60分钟课堂脚本: 按「5min导入→10min演示→25min实践→10min展示→10min总结反思」分配时间，写出每段的教师话术要点和预期学生反应；inputs: 素材包，outputs: 教案文档。
  6. 制定评价标准: 设计3维度量规（功能完成度/创意度/代码可读性），每维度3级描述；inputs: 教学目标，outputs: 评价量规表。
  - 每步「如果跳过会发生什么」: 跳过step2→课堂节奏失控，学生认知负荷超载；跳过step3→学生无从下手，卡在空白屏幕；跳过step6→展示环节流于表扬，失去诊断性评价机会。
- **资深路径**:
  - skip: 跳过 step 2 的知识点拆解文档，直接基于多年经验在脑中完成认知阶梯设计。
  - optimize: 优化 step 3，使用 Remix 机制批量生成差异化 starter，适配不同速度学生；同时优化 step 5，将固定脚本改为「节点+即兴」模式，保留灵活性。
  - add: 额外增加"课后作品上传社区"环节，学生互评3份其他同学作品，形成学习闭环 (evidence: [T03-S001])。
- **近期变化** (近12个月): AI辅助备课工具（如 ChatGPT for Teachers, evidence: [T03-S014]）显著缩短 step 4-5 耗时；Google CS First 提供 1000+ 配套视频，资源准备门槛降低。Scratch 3.0 扩展功能（Text-to-Speech、翻译积木）使课程主题更多样。
- **典型耗时**: 入门 3-4小时 / 资深 1-1.5小时。
- **关键工具**: scratch.mit.edu, ScratchEd Creative Computing Guide, Google CS First, Canva(PPT)。
- **常见失败模式**: ①主题过于抽象孩子不感兴趣；②知识点过多一节课塞5个以上；③未准备 starter 导致学生30分钟盯着空白屏幕；④没有展示环节，作品无人看，学生动机下降。
- **来源**: [T03-S001] [T03-S002] [T03-S004]
- **Last_updated**: 2026-05-26
- **Decay risk**: low

### 2. 规划 Python 进阶课程体系

- **One-liner**: 为从 Scratch 毕业或直接入门的10-14岁学生设计24-48课时 Python 课程大纲及配套项目体系。
- **Trigger**: 学生 Scratch 能力达标（能独立完成3个以上原创项目）；或新生直接评估后进入文本编程。
- **Output**: 课程大纲文档（含课时分配）、每单元配套项目、知识点图谱、学生入学测评题。
- **入门 SOP**:
  1. 学情评估与路径选择: 测评学生现有编程基础（Scratch经验、数学水平、兴趣方向），选择课程路径——路径A: turtle→pygame→web基础；路径B: turtle→数据分析→AI入门 (evidence: [T03-S009])；inputs: 学生档案，outputs: 路径决策+分班建议。
  2. 制定学习目标矩阵: 参照 CSTA 标准 (evidence: [T03-S004]) 为每个路径制定「知识/技能/理解」三层目标，时间跨度24课时（基础班）或48课时（进阶班）；inputs: 路径决策，outputs: 目标矩阵表。
  3. 拆解单元与项目主题: 每8课时为一个单元，每单元围绕一个驱动项目（如"制作贪吃蛇游戏""抓取天气数据绘图"），确保语法知识在项目中自然出现；inputs: 目标矩阵，outputs: 单元大纲草稿。
  4. 选择工具与环境: 基础阶段用 Thonny/IDLE 降低配置门槛，进阶阶段迁移至 VS Code 或 Replit (evidence: [T03-S009])；写出环境配置 SOP；inputs: 路径，outputs: 工具选型文档+安装脚本。
  5. 设计阶段性里程碑评估: 每单元末设置一个"展示项目"，期末设计综合项目；制定代码审查清单（变量命名/注释/模块化）；inputs: 单元大纲，outputs: 评价量规+期末项目说明书。
  6. 审核与试讲调整: 找1-2名同行教师审阅大纲，选1个单元做试讲，根据课堂反馈修订；inputs: 完整大纲，outputs: 修订版大纲v1.0。
  - 每步「如果跳过会发生什么」: 跳过step1→路径不匹配，学生挫败感高；跳过step3→课程变成知识点堆砌，学生不知道学了有什么用；跳过step6→上线后才发现进度估算偏差30%以上。
- **资深路径**:
  - skip: 跳过 step 6 的正式试讲，凭已有的多届迭代经验直接上线，在教学过程中动态调整。
  - optimize: 优化 step 1，使用在线自适应测评工具（如 Codingal 内置评估）替代人工测评；优化 step 3，将项目主题设计为可差异化的开放选题，高水平学生可选挑战版本。
  - add: 额外增加"家长版课程地图"，每两周发送学生进度简报，降低家长焦虑，提升续费率。
- **近期变化** (近12个月): AI辅助编程工具（GitHub Copilot、Replit AI）进入教学环境，课程需增加"与AI协作编程"模块和"识别AI错误"训练 (evidence: [T03-S010])；同时 Python 3.12+ 性能提升使树莓派课堂环境更稳定。
- **典型耗时**: 入门 2-3周 / 资深 3-5天。
- **关键工具**: Python.org, Replit, Thonny, VS Code, Codingal/Kodland平台, CSTA Standards。
- **常见失败模式**: ①过早引入 OOP 导致10岁学生大量流失；②项目主题成人化（股票分析），学生无共鸣；③忽略从 Scratch 图形化到文本编程的心理落差期（约2-4节课）；④环境配置课用掉了2节课，进度全面滞后。
- **来源**: [T03-S004] [T03-S009] [T03-S010]
- **Last_updated**: 2026-05-26
- **Decay risk**: medium

### 3. 设计信奥赛训练计划

- **One-liner**: 为决定参加 CSP/NOIP/NOI 竞赛路线的12-18岁学生制定12-24个月分阶段训练计划。
- **Trigger**: 学生或家长明确表达竞赛目标；教师发现学生在常规课程中持续超前完成。
- **Output**: 分阶段训练大纲（知识点图谱+刷题量+阶段目标）、月度进度追踪表、模拟赛日历。
- **入门 SOP**:
  1. 基线评估与目标校准: 用 Codeforces Div.3 或 Luogu P1000-P1500 级题目评估当前水平，设定12个月目标（CSP-J初赛入围 / CSP-S提高组 / NOIP参赛资格）；inputs: 学生基础，outputs: 评估报告+目标设定。
  2. 绘制知识点图谱: 梳理竞赛知识体系——基础层（STL/排序/贪心/简单DP）→中级层（图论/高级DP/数学）→高级层（数据结构/字符串/计算几何）(evidence: [T03-S007])；inputs: 目标等级，outputs: 知识点依赖图。
  3. 制定阶段性刷题计划: 每阶段2-3个月，设定每周刷题量（初期10题/周，竞赛前30题/周），平台以 Luogu + Codeforces + 洛谷题单为主 (evidence: [T03-S006])；inputs: 知识图谱，outputs: 周计划表格。
  4. 加入模拟赛与比较训练: 每月至少1次模拟赛（USACO月赛、Codeforces Round、或自制模拟赛），赛后必须进行错题复盘；inputs: 刷题计划，outputs: 模拟赛日历+复盘模板。
  5. 构建薄弱点补强机制: 建立"错题本"数据库（题目/知识点/错因/AC代码），每两周集中攻克高频错因知识点；inputs: 错题记录，outputs: 补强专项练习集。
  6. 赛前6-8周冲刺: 停止新知识点学习，集中做历年真题+全真模拟，调整作息到与考试同步；inputs: 训练积累，outputs: 冲刺阶段方案。
  - 每步「如果跳过会发生什么」: 跳过step1→目标设定严重偏高或偏低；跳过step5→反复在相同知识点失分；跳过step6→考场心理崩溃，发挥失常。
- **资深路径**:
  - skip: 跳过 step 3 的周计划文档，资深教练凭经验在每次课前直接挑选适配题目，动态调整节奏。
  - optimize: 优化 step 2，使用在线知识图谱工具（如 USACO Guide 的依赖树 evidence: [T03-S006]）替代手动绘制，节省初期规划时间。
  - add: 额外增加心理韧性训练模块（面对WA/TLE的情绪管理）和赛前读题速度训练，对抗竞赛时间压力。
- **近期变化** (近12个月): CSP-J/S 2024改革取消初赛机考部分，全面改为笔试+上机，影响备赛比重 (evidence: [T03-S007])；NOI 2025增加Python语言选项但C++仍为主流；AI辅助代码生成工具在训练中争议增大，多数教练要求禁用。
- **典型耗时**: 入门 2-4周规划 / 资深 1周规划。
- **关键工具**: Luogu, Codeforces, USACO Guide, 洛谷题单, VSCode + GCC, 本地OJ。
- **常见失败模式**: ①急于学高级算法，跳过基础排序/暴力枚举；②刷题数量多但缺乏复盘，错题反复犯；③只刷简单题建立舒适区，没有主动触碰困难题；④考前1-2周引入新知识点，扰乱心理状态。
- **来源**: [T03-S006] [T03-S007]
- **Last_updated**: 2026-05-26
- **Decay risk**: medium

### 4. 设计机器人/创客课程

- **One-liner**: 为8-15岁学生设计含硬件选型+课程大纲+耗材清单的机器人/创客完整课程体系。
- **Trigger**: 机构决定增设硬件课程线；学校STEAM实验室筹建；比赛（WRO/FLL/VEX）备赛需要。
- **Output**: 硬件选型报告、分级课程大纲（初/中/高）、耗材采购清单、维修维护SOP。
- **入门 SOP**:
  1. 确定课程目标与受众: 确认年龄段（8-10岁/11-13岁/14-15岁）、课程性质（兴趣班/竞赛班/校本课程）、预算区间；inputs: 机构定位，outputs: 需求说明书。
  2. 硬件选型决策: 初级选 LEGO SPIKE Essential 或 micro:bit（门槛低、无需焊接）；中级选 LEGO Mindstorms EV3 / mBot（编程深度更高）；高级选 Arduino / Raspberry Pi（开放平台）(evidence: [T03-S005], [T03-S013])；inputs: 需求说明书+预算，outputs: 硬件选型报告。
  3. 配套软件与编程环境选型: 初级对应 Scratch 图形化扩展或 MakeCode；中级对应 EV3 Lab 或 Python；高级对应 C++/MicroPython (evidence: [T03-S013])；inputs: 硬件选型，outputs: 软件配置说明。
  4. 设计分级课程大纲: 参照 LEGO Education 课程框架 (evidence: [T03-S005])，每级8-12课时，课时结构为「工程挑战→搭建→编程→测试→优化→展示」；inputs: 硬件+软件方案，outputs: 分级大纲文档。
  5. 制定耗材采购与管理计划: 列出每班次耗材（备用零件/线材/传感器），设计零件归还检查表，计算每生每课时耗材成本；inputs: 大纲，outputs: 采购清单+管理SOP。
  6. 试运行与迭代: 选1个班级先行运行2-3课时，记录零件损坏率、学生卡点、进度达成率，修订后全面铺开；inputs: 大纲+采购完成，outputs: 试运行报告+修订大纲v1.0。
  - 每步「如果跳过会发生什么」: 跳过step2的充分调研→买了孩子玩不了或教师不会教的硬件；跳过step5→课堂零件管理混乱，半学期后核心零件严重缺失；跳过step6→全面铺开后才发现进度严重不匹配。
- **资深路径**:
  - skip: 跳过 step 6 的正式试运行，资深教师基于5年以上硬件教学经验直接判断大纲可行性。
  - optimize: 优化 step 5，与本地供应商签订"零件补货协议"（损坏零件按件补购），降低备用库存压力；优化 step 4，将工程挑战设计为开放性任务（"让机器人走最短路径"），而非固定答案。
  - add: 额外增加"家长工作坊"（学期末让家长体验搭建），强化家长认可度与续费意愿。
- **近期变化** (近12个月): micro:bit V2 和 LEGO SPIKE Prime 成为替代 EV3 的主流选择（EV3逐渐停产）；AI视觉模块（如 Huskylens）开始进入中小学创客课堂；WRO 2024规则更新要求机器人具备自主判断能力，影响备赛训练重点。
- **典型耗时**: 入门 2-3周 / 资深 1周。
- **关键工具**: education.lego.com, micro:bit MakeCode, mBlock, Arduino IDE, FLL/WRO官方规则文件。
- **常见失败模式**: ①选型超预算又无法降级，课程被迫砍半；②硬件平台太难，教师自身也不熟练，课堂失控；③零件管理混乱，课堂30%时间用于找零件；④大班（30人+）机器人课无助教支撑，学生等待时间过长。
- **来源**: [T03-S005] [T03-S013]
- **Last_updated**: 2026-05-26
- **Decay risk**: low

### 5. 新建少儿编程培训机构 (从0到1)

- **One-liner**: 从创业决策到首期开课，完成选址/装修/课程/师资/招生/系统全链条搭建，约3-6个月。
- **Trigger**: 个人或团队决定创业或加盟；教育集团决定开设新校区。
- **Output**: 开业就绪的培训机构（教室+课程体系+教师+学生）。
- **入门 SOP**:
  1. 模式决策（自营 vs 加盟）: 评估资金（加盟需50-200万，自营15-80万）、课程研发能力、品牌资源；加盟路径风险低但利润分成约30-50%；inputs: 资金+经验，outputs: 商业模式决策书。
  2. 选址与市场调研: 优先学校周边500米或社区商圈，核查竞对密度（3公里内同类机构数）、目标客户家庭收入水平、周边小学/初中招生规模；inputs: 模式决策，outputs: 选址分析报告+意向合同。
  3. 场地改造与教室设计: 按每生3平米计算容量，配置稳定网络（百兆独享）、备用电源、每台电脑独立UPS；教室布局参照小组学习模式（4人一组）；inputs: 租约，outputs: 装修图纸+验收清单。
  4. 课程体系搭建: 根据目标年龄段选定课程（Scratch/Python/机器人），采购或自研教材，制定教学进度标准；inputs: 市场调研结果，outputs: 课程目录+教材版本。
  5. 师资招募与培训: 发布招聘（应届CS师范生+有编程背景的年轻人），完成岗前培训（技术+教学法+模拟授课考核），合格后上岗；参考 Code.org 教师培训框架 (evidence: [T03-S003])；inputs: 课程体系，outputs: 教师团队+培训档案。
  6. 运营系统配置: 搭建学员管理系统（CRM/排课/缴费/家长通知）、准备收据发票、建立微信家长群SOP；inputs: 人员到位，outputs: 运营流程手册。
  7. 首期招生: 举办免费体验课（目标50人进场，转化10-20名正式学员），配合朋友圈/小红书/社区地推；inputs: 场地+师资就绪，outputs: 首期学员名单+营收预期。
  - 每步「如果跳过会发生什么」: 跳过step2的竞对调研→定价与市场错位；跳过step6→开业第二周账单混乱，退费纠纷无记录；跳过step5的模拟授课考核→教师第一堂课失控，首批学员流失。
- **资深路径**:
  - skip: 跳过 step 1 的详细可行性分析，有连锁运营经验者凭选址数据直接判断可行性。
  - optimize: 优化 step 7，将体验课改为"亲子工作坊"形式，家长参与度提升，当天转化率提高15-25%；优化 step 3，采用标准化装修包（连锁品牌统一材料清单），压缩装修周期4-6周至2-3周。
  - add: 额外增加政策合规环节（营业执照/办学许可证/消防验收），避免开业后被责令整改停业（国内监管趋严，2024年后更严格）。
- **近期变化** (近12个月): 中国"双减"政策持续影响，学科类培训受限但编程（STEAM）政策利好；但2024年起部分省市要求编程培训机构申请专项备案；AI课程（ChatGPT入门/AI绘画）成为新卖点，拉高客单价；线上+线下混合模式普及，降低对固定场地的依赖。
- **典型耗时**: 入门 3-6个月 / 资深 6-8周。
- **关键工具**: 学员管理SaaS（校管家/爱学习）, 腾讯会议/Zoom(线上课), 微信生态(招生), 招聘平台(BOSS直聘)。
- **常见失败模式**: ①选址人流量低估，开业后日均进店不足10人；②师资不稳定，首月开课就换教师；③课程定价参考竞对最低价，利润空间不足维持运营；④忽视政策合规，被下架整改。
- **来源**: [T03-S003] [T03-S004]
- **Last_updated**: 2026-05-26
- **Decay risk**: low

### 6. 招生与转化流程

- **One-liner**: 在新学期/寒暑假前8-12周，通过公开课→体验课→正课的漏斗将潜在家庭转化为付费学员。
- **Trigger**: 新学期开学前2个月；寒暑假前4-6周；机构新校区筹备期。
- **Output**: 目标学员数量（正课付费转化）、家长群建设、课程排班计划。
- **入门 SOP**:
  1. 制定招生目标与漏斗预期: 设定目标学员数（如30人/班×2班=60人），逆推漏斗各层——付费学员60人需体验课120人需公开课进场300人；inputs: 机构容量，outputs: 招生目标表。
  2. 设计引流活动（公开课/线上直播): 设计30-45分钟免费公开课（如"孩子用Scratch做会动的生日贺卡"），海报/短视频在微信/抖音/小红书投放，目标触达500-1000人；inputs: 招生目标，outputs: 活动策划书+物料。
  3. 线索收集与初步筛选: 公开课后收集家长微信/手机，24小时内跟进添加，发送课程介绍+体验课预约链接；inputs: 公开课出席名单，outputs: 线索数据库。
  4. 体验课设计与交付: 设计60-90分钟付费或免费体验课（强调"孩子现场做出作品"），课中家长旁听，课后销售顾问主动沟通；对应教育招生漏斗最关键节点 (evidence: [T03-S004])；inputs: 线索，outputs: 体验课出席学生+家长意向记录。
  5. 家长决策沟通与异议处理: 整理常见家长疑虑（孩子坐不住/价格/学了有用吗），准备标准话术；1对1沟通优于群发；inputs: 家长意向记录，outputs: 意向家长跟进列表。
  6. 正式转化与入学手续: 签协议/缴费/排班，同时邀请家长加入专属微信群，建立后续粘性；inputs: 意向确认，outputs: 学员花名册+缴费凭证。
  - 每步「如果跳过会发生什么」: 跳过step1的漏斗预期→中途才发现体验课人数不够，临时加场手忙脚乱；跳过step5的1对1沟通→转化率从30%降至10%以下；跳过step6的入群环节→学员开课前流失率升高。
- **资深路径**:
  - skip: 跳过线下地推，纯依赖老学员口碑转介绍（转介绍返学费优惠），资深机构老学员口碑漏斗可覆盖70%+招生需求。
  - optimize: 优化 step 2，将引流课内容标准化为"爆款视频脚本"（固定话术+互动设计），实现低成本复制到多个销售渠道；优化 step 4，增加"孩子作品当场分享家长群"环节，提升家长感知价值。
  - add: 额外增加 NPS 调研（体验课后1周发送问卷），追踪转化失败原因并建立改进机制。
- **近期变化** (近12个月): 抖音/视频号短视频招生效果持续优于传统地推（2024年数据显示短视频线索成本降低40%）；AI体验课（"让孩子和AI一起做游戏"）成为新爆款引流内容；家长决策周期从3-7天缩短至体验课当天（高质量体验课+限时优惠组合）。
- **典型耗时**: 入门 8-12周/季 / 资深 4-6周/季。
- **关键工具**: 微信/企业微信, 抖音/视频号, 问卷星, 学员CRM系统, 腾讯会议(线上体验课)。
- **常见失败模式**: ①体验课太像"正课"，家长感觉在试听，没有"孩子完成作品"的峰值体验；②线索收集后48小时无跟进，转化窗口关闭；③价格策略混乱，体验课当天给优惠后正式学员要求追溯；④公开课场次过多，教师精力耗尽，体验课质量下降。
- **来源**: [T03-S004] [T03-S003]
- **Last_updated**: 2026-05-26
- **Decay risk**: medium

### 7. 教师培训与认证

- **One-liner**: 将新入职教师或课程升级背景下的在职教师，培养至能独立交付标准课程质量的上岗状态。
- **Trigger**: 新教师入职（岗前培训）；课程体系升级（如增加AI课程线）；教学质量评估发现短板。
- **Output**: 通过考核的上岗教师（技术能力+教学法能力双达标）、培训档案、后续督导计划。
- **入门 SOP**:
  1. 基线评估: 测评新教师编程技术水平（完成指定项目）+ 教育理念了解（开放式问答）；inputs: 招聘简历+面试记录，outputs: 能力基线报告。
  2. 技术培训（产品知识）: 系统学习机构课程体系——跟着学生视角完整上一遍所有课程，掌握每节课的知识点、常见学生困惑点和调试方法；参考 Code.org 教师PD框架 (evidence: [T03-S003])；inputs: 课程教材，outputs: 课程掌握自查表（每节课打分）。
  3. 教学法培训: 学习儿童认知发展特点（Piaget/Vygotsky 基础）、提问技巧（苏格拉底式 vs 直接提示）、课堂管理技巧；观摩资深教师至少3次课 (evidence: [T03-S011])；inputs: 培训材料+观摩安排，outputs: 教学观摩记录+反思日志。
  4. 模拟授课: 对培训师（扮演学生）完整授课1-2节，接受实时反馈；重点评估：问题引导能力、时间控制、学生卡点处理；inputs: 观摩完成，outputs: 模拟授课评估表。
  5. 试讲（真实学生）: 在督导教师陪同下对真实学生授课2-4节，督导填写详细观察表；inputs: 模拟授课通过，outputs: 督导评估报告。
  6. 上岗认证与持续发展: 综合技术+模拟+试讲三项评分，达标即颁发内部上岗证；制定后续30-60-90天督导计划，每月1次随堂观察；inputs: 所有评估记录，outputs: 上岗证书+督导日历。
  - 每步「如果跳过会发生什么」: 跳过step3的教学法培训→教师以大学助教方式讲课，6-9岁学生完全失去兴趣；跳过step5的真实试讲→首次正式上课出现重大问题，引发家长投诉。
- **资深路径**:
  - skip: 跳过 step 2 的逐课自学环节，资深教师可直接跳到模拟授课，用实战暴露短板效率更高。
  - optimize: 优化 step 3，引入"同伴学习"模式（新教师两两结对互相评课），减少对培训师资源的占用；优化 step 4，录制模拟授课视频供自我复盘。
  - add: 额外增加外部认证路径（如 CSTA 认证、CodeHS PD 证书 evidence: [T03-S011]），提升教师职业发展感，降低离职率。
- **近期变化** (近12个月): AI工具使用培训成为2025年教师培训必修模块（ChatGPT辅助备课、AI工具在课堂中的伦理使用 evidence: [T03-S014]）；Google推出"Generative AI for Educators"2小时免费认证课，已成为行业标配培训资源；线上培训（直播+录播）降低了多校区机构的培训成本。
- **典型耗时**: 入门 4-8周 / 资深 2-3周。
- **关键工具**: Code.org Teacher PD, CodeHS PD, Google AI for Educators, Zoom/腾讯会议(远程培训), 录屏工具(Loom)。
- **常见失败模式**: ①培训只有技术没有教学法，教师上课变成"技术讲解"而非"学习引导"；②试讲环节省略或走形式；③培训结束后无持续督导，教学质量快速回归基线；④培训内容与实际授课课程脱节（培训A课程，实际教B课程）。
- **来源**: [T03-S003] [T03-S011] [T03-S014]
- **Last_updated**: 2026-05-26
- **Decay risk**: medium

### 8. 一堂编程课的交付流程

- **One-liner**: 从课前10分钟准备到课后反思，完整执行一节60-90分钟编程课，产出学生项目+教学日志。
- **Trigger**: 每次正式上课前（每节课常规执行，非一次性流程）。
- **Output**: 学生完成或接近完成的编程项目、教师课堂反思记录、家长沟通消息（若有特殊情况）。
- **入门 SOP**:
  1. 课前准备（-10分钟）: 检查所有电脑开机正常、Scratch/Python环境可用、starter 文件已放共享文件夹；打印"今日任务卡"；inputs: 教案，outputs: 教室就绪确认清单。
  2. 复习导入（5分钟）: 提问上节课知识点（2-3个问题），快速唤醒记忆；用"今天我们要做什么"激发期待；inputs: 上节课记录，outputs: 学生注意力集中。
  3. 新知讲解与演示（10分钟）: 教师在投影上演示新功能/新概念，用"猜猜会发生什么"等互动提问保持参与度；每次演示不超过3个新知识点 (evidence: [T03-S002])；inputs: 教案，outputs: 学生理解新知。
  4. 实践操作（25-35分钟）: 学生在自己电脑上完成任务，教师巡场—重点观察「卡住超过3分钟」的学生，优先用提问引导而非直接给答案；inputs: starter文件+任务卡，outputs: 学生作品（进行中）。
  5. 展示分享（10分钟）: 邀请2-3名学生展示作品，全班给出"1个亮点+1个建议"，教师总结今日核心收获；inputs: 学生作品，outputs: 同伴反馈+学生成就感。
  6. 总结与布置拓展（5分钟）: 复述今日3个关键词，布置可选拓展任务（分级：基础/挑战），提醒下次课带什么；inputs: 课堂进展，outputs: 拓展任务说明。
  7. 课后反思记录（15分钟内）: 记录：①哪个知识点卡了最多人②时间分配是否合理③需要下次跟进的学生名单；inputs: 课堂观察，outputs: 教学日志条目。
  - 每步「如果跳过会发生什么」: 跳过step2→学生前10分钟需要重新进入状态，浪费15-20分钟；跳过step5→学生感觉"写完就结束了"，缺乏成就感和社交学习；跳过step7→相同问题下周还会出现，教师无法积累教学改进。
- **资深路径**:
  - skip: 跳过逐字教案，资深教师用"关键节点清单"（一张A4纸）代替完整教案，课堂即兴发挥占40%。
  - optimize: 优化 step 4，部署"同伴辅导"机制（进度快的学生帮助卡住的同学），教师从"消防员"变为"协调者"；优化 step 5，建立"作品画廊"模式（课末所有人投影展示20秒）替代抽样展示。
  - add: 额外增加课后家长推送（微信小程序自动推送学生今日作品截图），增强家长参与感与续费粘性 (evidence: [T03-S012])。
- **近期变化** (近12个月): AI助手（GitHub Copilot、Replit AI）进入学生电脑，需要在实践环节设定明确的"用AI辅助"vs"独立完成"规则；Code.org 推出 AI for Oceans 等新课程模块，10分钟即插即用；在线课堂（双师模式）标准化交付流程被更多机构采用。
- **典型耗时**: 入门 90min课+30min课前课后 / 资深 60min课+15min课前课后。
- **关键工具**: Scratch/Python/Replit（课堂环境）, 投影/交互白板, 共享文件夹(腾讯文档/Google Drive), 教学日志模板, 计时器App。
- **常见失败模式**: ①演示时间过长压缩实践时间（学生只看没有做）；②巡场时只帮最快的学生增加挑战，忽略中间大多数；③没有展示环节，全程只有教师在讲；④课后不记录，下次课重复相同错误。
- **来源**: [T03-S002] [T03-S012] [T03-S003]
- **Last_updated**: 2026-05-26
- **Decay risk**: low

### 9. 编程作品评价与展示

- **One-liner**: 在单元结束/期末/比赛前，组织学生作品的系统性评价和公开展示，产出作品集和评价报告。
- **Trigger**: 每个课程单元结束；学期末；参加比赛前；机构开放日/家长会。
- **Output**: 学生作品集（可分享链接/视频）、评价报告（个人维度）、展示活动记录。
- **入门 SOP**:
  1. 确定展示形式与受众: 选择展示形式——班内互评/家长开放日/线上作品墙/比赛提交；确定受众影响评价标准的制定；inputs: 课程节点+机构日历，outputs: 展示方案。
  2. 设计评价量规: 制定3-5维度评价标准（功能完成度/创意度/代码规范/展示表达/用户体验），每维度3级描述；量规需提前发给学生作为自评工具 (evidence: [T03-S012])；inputs: 教学目标，outputs: 评价量规表。
  3. 学生自评与同伴互评: 学生先完成自评表（3-5分钟），再与1-2名同学交换评价；教师收集所有自评表；inputs: 量规表，outputs: 自评+互评记录。
  4. 作品集整理与打包: 指导学生将作品导出（Scratch分享链接/Python截图+代码/视频录制），整理到统一平台（班级作品墙）；低年龄段可用 ScratchEd 社区，高年级用 GitHub Pages (evidence: [T03-S001])；inputs: 学生作品，outputs: 可访问的作品集链接。
  5. 公开展示活动: 安排每生2-3分钟"作品介绍"（说明：做了什么/怎么做到的/下一步想改什么）；准备评委打分表（若有外部评委）；inputs: 作品集，outputs: 展示活动记录+观众反馈。
  6. 教师综合评价与反馈: 综合自评/互评/展示表现，为每名学生写1段定性反馈（50-100字），发给家长；inputs: 所有评价记录，outputs: 个人评价报告+家长通知。
  - 每步「如果跳过会发生什么」: 跳过step2→展示变成随意表扬，学生不知道如何改进；跳过step3的自评→评价主权在教师，学生缺乏元认知发展；跳过step5的公开展示→作品停留在"完成作业"层面，没有真实受众驱动的动力。
- **资深路径**:
  - skip: 跳过纸质评价量规，资深教师在展示现场即兴给出口头结构化反馈，效果等同但对教师经验依赖度高。
  - optimize: 优化 step 4，使用自动化作品墙工具（OpenWater/SkoolOfCode等）替代手动整理，节省1-2小时行政时间 (evidence: [T03-S012])；优化 step 6，将定性反馈模板化，高效产出个性化报告。
  - add: 额外增加"最佳作品评选"（班内投票）和颁奖仪式，强化学生荣誉感；增加作品存档到机构官网/社交媒体作为招生素材。
- **近期变化** (近12个月): AI生成代码/设计开始出现在学生作品中，评价量规需增加"原创性说明"维度；视频展示（录屏+讲解）替代现场演示成为线上班级主要展示形式；GitHub Pages 和 Replit 的免费部署功能使"真实发布"门槛大幅降低。
- **典型耗时**: 入门 1-2周（设计+组织+执行）/ 资深 3-5天。
- **关键工具**: Scratch社区（作品分享）, GitHub Pages, SkoolOfCode, OpenWater（评选平台）, 腾讯问卷（自评互评）, OBS/Loom（录屏展示）。
- **常见失败模式**: ①评价量规在展示当天才发给学生，学生无法按标准完善作品；②展示时间安排过紧（30人班级每人3分钟=1.5小时），家长等待疲惫；③缺乏外部受众，班内展示流于形式；④评价结果不传达给家长，家长对孩子学了什么一无所知。
- **来源**: [T03-S001] [T03-S012]
- **Last_updated**: 2026-05-26
- **Decay risk**: low

### 10. 备战 NOIP/NOI 冲刺

- **One-liner**: 在赛前3-6个月进入冲刺阶段，停止新知学习，集中强化真题训练、模拟赛、心理调适，最大化竞赛发挥。
- **Trigger**: 距离 CSP-J/S 或 NOIP 考试约3-6个月；学生已完成系统知识图谱学习（达到目标等级知识覆盖80%+）。
- **Output**: 冲刺训练计划（按周排列）、历年真题通过率统计、模拟赛成绩记录、考前心理准备方案。
- **入门 SOP**:
  1. 现状诊断与能力摸底: 用3-5套历年真题（CSP-J近3年/NOIP近5年）计时完成，统计每题AC率和时间分布；识别高频失分知识点和"会但不稳"的知识点；inputs: 历年真题，outputs: 能力摸底报告。
  2. 制定冲刺优先级: 将知识点分为3类——A类（稳定可拿分，保持熟练度）/ B类（会但不稳，重点强化）/ C类（从未掌握，赛前放弃，避免分散精力）；inputs: 摸底报告，outputs: 知识点优先级矩阵。
  3. 规划每周训练节奏: 每周安排：3次专项练习（各1-2小时）+ 1次全真模拟赛（完整时长+真实评测环境）+ 1次错题复盘；参考 USACO Guide 的冲刺节奏建议 (evidence: [T03-S006])；inputs: 优先级矩阵，outputs: 周训练日历。
  4. 全真模拟赛演练: 每次模拟赛严格执行考试规则（全程断网、限时、不查资料），赛后立即复盘——"如果不卡在X，能多拿Y分"；inputs: 训练日历，outputs: 模拟赛成绩记录+复盘日志。
  5. 读题与策略训练: 训练"5分钟读完4题并判断难易顺序"，培养先做确定分、后攻部分分的答题策略；inputs: 真题集，outputs: 读题速度基准数据。
  6. 赛前2周调整与心理准备: 停止新题训练，只做已会题型的"热身练习"；调整作息到与考试时间吻合；教师/家长进行考前谈话（正常发挥 > 超常发挥）；inputs: 冲刺训练完成，outputs: 考前状态评估+考试注意事项清单。
  - 每步「如果跳过会发生什么」: 跳过step2的C类放弃决策→考前仍在学新知识点，焦虑升高、B类强化不足；跳过step5的读题策略训练→考场因读题慢损失20-30分钟；跳过step6的心理准备→考场压力性失误。
- **资深路径**:
  - skip: 跳过 step 1 的系统摸底，资深教练已掌握该生知识图谱，直接制定冲刺计划。
  - optimize: 优化 step 4，引入"对拍"机制（两名学生用不同算法解同一题，交叉验证答案），比对答案差异比单纯AC率更有诊断价值；优化 step 3，赛前最后4周将模拟赛频率从每周1次提升至每周2次。
  - add: 额外增加"赛后心理恢复计划"——无论成绩如何，赛后1周安排轻松活动，避免竞赛压力积累影响下一赛季备赛。
- **近期变化** (近12个月): CSP-J 2024引入更多"工程题"（给定代码框架填写关键函数），读题和理解代码速度成为新关键技能 (evidence: [T03-S007])；Luogu 题目标签系统升级，使精准专项练习更容易；AI辅助代码调试（Copilot等）在训练中讨论增多，但大多数省赛明确禁用；信奥赛参赛人数持续增长（全国约10万人/年），竞争烈度上升。
- **典型耗时**: 入门 6个月冲刺 / 资深 3个月冲刺。
- **关键工具**: Luogu, Codeforces, USACO Guide, 本地评测环境(Dev-C++/VSCode+GCC), 历年真题合集, 计时器。
- **常见失败模式**: ①冲刺期仍在学新算法（贪心不会就临时补），导致基础知识点退化；②模拟赛不断网，失去真实考场压力训练效果；③只追求AC数量，忽略"稳定拿到确定分"的策略训练；④赛前焦虑失眠，未提前调整作息。
- **来源**: [T03-S006] [T03-S007]
- **Last_updated**: 2026-05-26
- **Decay risk**: medium

## Phase 2 提炼提示

**反复出现≥3个workflow都包含的步骤** (候选playbook):
1. **基线评估/诊断** — 出现于 WF1（学情）、WF2（学情测评）、WF3（能力摸底）、WF6（漏斗预期）、WF7（教师基线）、WF10（真题诊断）。核心模式：不做任何计划前先测量现状，产出"能力基线报告"。
2. **分级/差异化设计** — 出现于 WF1（认知阶梯）、WF2（路径A/B选择）、WF4（初/中/高硬件选型）、WF8（任务卡分基础/挑战）、WF9（量规3级）。核心模式：任何交付物都要设计至少2个级别满足不同速度学习者。
3. **试运行/模拟/试讲** — 出现于 WF1（试讲调整）、WF4（试运行1班）、WF7（模拟授课→试讲）、WF10（全真模拟赛）。核心模式：在全面铺开前做小规模验证，用真实反馈修订计划。
4. **公开展示/成果输出** — 出现于 WF1（学生展示）、WF8（展示分享环节）、WF9（公开展示活动）、WF5（体验课作品展示）。核心模式：任何学习周期结束都需要一个"真实受众"的作品展示锚点。
5. **课后/赛后复盘记录** — 出现于 WF3（错题复盘）、WF8（教学日志）、WF9（评价报告）、WF10（模拟赛复盘日志）。核心模式：执行结束后必须有结构化记录，驱动下一轮改进。

**入门SOP和资深路径最大差距** (候选心智模型):
- **文档化 vs 内化**: 入门教师依赖完整文档（教案、量规、清单），资深教师将大多数决策内化为直觉，文档退化为"关键节点清单"。
- **顺序执行 vs 并发/跳步**: 入门必须严格按步骤顺序，资深可同时处理多个步骤或直接跳过已内化的环节。
- **消防员 vs 协调者**: 入门教师在课堂是"问题解决者"（学生卡住→教师去解决），资深教师是"环境设计者"（提前设计让学生互相解决的机制）。
- **结果导向 vs 信号导向**: 入门看"完成了多少题/课"，资深盯"哪个知识点反复失分/哪类学生流失"等信号。

**近期工作流变化的根本原因**:
1. **生成式AI工具普及** (2024-2025): 影响几乎所有10个workflow——教师备课加速（WF1/2/7）、学生AI辅助编程引发评价标准重构（WF9）、竞赛环节禁用AI引发训练策略调整（WF10）。
2. **家长期望升级**: 家长不再满足于"孩子学了编程"，要求看到真实作品和升学/竞赛价值，驱动展示（WF9）和竞赛路线（WF3/WF10）需求增长。
3. **中国政策环境**: "双减"后编程/STEAM政策利好，但合规要求提升（WF5需备案）；CSP考试改革（WF3/WF10影响备赛策略）。
4. **硬件平台迭代**: EV3退场、SPIKE Prime/micro:bit主流化，机器人课程（WF4）选型逻辑已发生结构性改变。

**冷僻/信号薄弱**:
- 编程课程的"情感/动机"设计维度（学生保持学习动力的机制）在所有公开资料中信号极弱，几乎没有系统化SOP。
- 农村/低收入地区少儿编程教育路径（低配硬件、不稳定网络条件下的课程交付）几乎无公开方法论。
- 竞赛后段（NOI→集训→IOI国际赛）的训练方法论极少公开，信号薄弱。
- 编程教育的"学员流失/退费"主动干预SOP（而非被动响应）几乎没有标准化方法论。
